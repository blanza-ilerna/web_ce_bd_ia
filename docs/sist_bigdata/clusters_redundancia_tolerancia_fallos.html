Redundancia y Tolerancia a Fallos 🛡️

Garantizando la continuidad operativa en sistemas distribuidos

En un clúster, el fallo no es una posibilidad, es una certeza para la que el sistema debe estar siempre preparado.

El principio de la persistencia

La redundancia es el pilar que sostiene la fiabilidad de un clúster. Como en cualquier red distribuida existe el riesgo de que una máquina falle, el diseño debe garantizar que el servicio continúe sin interrupciones ni pérdida de información.

¿Cómo se implementa la redundancia?

No se trata solo de copiar datos, sino de asegurar que todo el ecosistema sea resiliente mediante dos vías principales:

💾

Replicación de Datos

Cada pieza de información se almacena en varios nodos físicos. Si un disco falla, el sistema recupera la copia idéntica desde otro nodo sin que el usuario lo perciba.

🔄

Replicación de Procesos

Las tareas críticas disponen de nodos de respaldo. Si una ejecución se detiene por error, el sistema reinicia el proceso automáticamente en otra máquina disponible.

El Latido del Sistema: Heartbeat 💓

¿Cómo sabe el clúster que un nodo ha muerto? Mediante señales periódicas llamadas heartbeats.

  • Cada nodo de trabajo envía un "estoy vivo" al maestro de forma constante.
  • Si el maestro deja de recibir este latido durante un tiempo predefinido, marca al nodo como fallido.
  • Automáticamente, el sistema redistribuye la carga de ese nodo entre el resto de la flota.

📍 Punto Clave

La redundancia no es un desperdicio de recursos (almacenar lo mismo varias veces), sino un seguro de vida para el dato. En Big Data, es preferible pagar por el espacio extra de réplica que arriesgarse a una pérdida irreversible.
